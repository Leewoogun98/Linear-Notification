# Linear 커스텀 알림 데스크탑 앱 — 설계 문서

작성일: 2026-06-24

## 1. 목적 / 배경

Linear의 기본 데스크탑 알림은 화면 우상단에 떠서 눈에 잘 띄지 않는다. 사용자가
**직접 설정한 규칙에 맞는 알림만** 화면 **정중앙**에 크게 띄워서 놓치지 않게 하는
것이 목표다.

- 실시간성을 위해 **폴링이 아닌 webhook 방식** 사용
- 윈도우/맥 모두 배포 가능해야 하므로 **Electron** 사용
- 사용 범위: **나 혼자** (단일 Linear 워크스페이스)

## 2. 전체 구조

```
                    (1) 이벤트 발생
   Linear 워크스페이스
        │
        │ (2) Webhook POST (Linear-Signature 서명됨)
        ▼
   ┌─────────────────────────────┐
   │  Cloudflare Worker (relay)  │
   │  - 서명 검증                 │
   │  - Durable Object로 전달     │
   │  ┌───────────────────────┐  │
   │  │  Relay Durable Object │  │  ← WebSocket 연결 유지 + 최근 이벤트 짧게 버퍼
   │  └───────────────────────┘  │
   └──────────────┬──────────────┘
                  │ (3) WebSocket push (실시간)
                  ▼
   ┌─────────────────────────────┐
   │   Electron 데스크탑 앱       │
   │  - WS 클라이언트 (자동 재연결)│
   │  - 규칙 엔진 (매칭 판단)      │
   │  - 매칭되면 → 중앙 알림창     │
   │  - 트레이 아이콘 / 설정창     │
   └─────────────────────────────┘
```

**핵심 설계 판단:** 규칙 평가(필터링)는 Cloudflare가 아니라 **데스크탑 앱에서**
수행한다.

- 규칙을 바꿔도 서버를 재배포할 필요가 없다.
- 규칙/설정이 로컬에 있어 편집이 쉽다.
- relay는 어떤 규칙이 있는지 알 필요 없이 "받아서 전달"만 한다 (책임 분리).

## 3. 컴포넌트

각 컴포넌트는 하나의 명확한 책임을 가지며 독립적으로 이해/테스트 가능하도록 구성한다.

### A. Relay (`LinearServer/`, Cloudflare Workers + TypeScript)

"relay"는 프레임워크가 아니라 **역할명(중계 서버)** 이다. Linear webhook을 받아
데스크탑 앱으로 그대로 전달하는 일만 한다. 비즈니스 로직 없음.

- `worker` — Linear webhook POST 수신, `Linear-Signature`(HMAC-SHA256) 검증,
  Durable Object로 전달. 데스크탑 앱의 WS 업그레이드 요청도 여기서 받아 토큰 검증 후
  Durable Object로 라우팅.
- `RelayDurableObject` — 데스크탑 앱의 WebSocket 연결 유지, 들어온 이벤트를 연결된
  앱에 fan-out, **최근 ~60초 이벤트 버퍼** 보관(앱이 잠깐 끊겼다 재연결하면 놓친
  이벤트 replay).
- 인증: 앱이 WS를 열 때 토큰 제시 → Worker가 검증. 단일 사용자이므로 공유 토큰 하나.

플랫폼 제약: Cloudflare Workers는 V8(JS/TS) 런타임이라 JVM(Spring 등)은 사용 불가.
relay는 TypeScript로 작성한다.

### B. 데스크탑 앱 (`LinearApp/`, Electron)

- `main` (메인 프로세스) — WS 클라이언트(지수 백오프 재연결), 트레이 아이콘, 창 관리,
  소리 재생, 설정 저장 조율.
- `rule-engine` — **순수 모듈**. `(이벤트, 규칙들) → { 매칭 여부, 표시 텍스트 }`.
  외부 의존성 없음 → 단위 테스트가 쉽다.
- `notification-window` — 화면 정중앙 알림 UI. 스택으로 쌓기, 5초 자동 사라짐,
  클릭하면 닫힘.
- `settings-window` — 규칙 편집 UI (이벤트 타입 + 필터 조합), 내 Linear user ID 및
  relay 접속 정보 입력.
- `config-store` — 규칙/설정/relay 토큰을 로컬에 저장 (electron-store 또는 JSON 파일).

## 4. 규칙 모델

한 규칙 = **이벤트 타입(들)** + **필터 조건(들, AND 결합)** → 매칭되면 중앙 알림.

- **이벤트 타입**: 이슈 생성/수정, 코멘트, 프로젝트, 프로젝트 업데이트 등 Linear
  webhook이 제공하는 타입.
- **필터**: 팀 / 프로젝트 / 라벨 / assignee / 나를 멘션 / 제목·본문 키워드.
- 예시:
  - `이벤트=코멘트 AND 나를 멘션` → 중앙 알림
  - `이벤트=이슈 AND 라벨=urgent` → 중앙 알림

"나를 멘션" 판별을 위해 설정창에서 내 Linear user ID를 한 번 입력받는다.

## 5. 알림(중앙 알림창) 동작

- **표시 내용**: Linear webhook payload가 제공하는 텍스트를 최대한 모두 표시(이슈
  제목/설명, 코멘트 본문, 행위자 등) — 사용자가 클릭 없이 바로 내용을 확인할 수 있게.
- **자동 사라짐**: 5초 후 자동으로 사라짐.
- **클릭**: 클릭하면 닫힘(사용자가 확인했다는 의미). 별도의 "Linear 열기"는 없음.
- **소리**: 알림 표시 시 소리 재생.
- **동시 도착**: 스택 형태로 쌓아서 표시.

## 6. 데이터 흐름 / 에러 처리

- **정상 흐름**: Linear → (서명된 webhook) → Worker 서명 검증 → Durable Object →
  WS push → 앱 규칙 평가 → 매칭 시 중앙 알림 + 소리 → 5초/클릭 시 닫힘.
- **앱이 꺼져 있을 때**: 그 사이 이벤트는 놓침(알림 성격상 허용).
- **짧은 끊김**: Durable Object의 60초 버퍼로 재연결 시 복구(replay).
- **WS 끊김**: 자동 재연결(지수 백오프), 재연결 시 버퍼 replay 수신.
- **서명 검증 실패**: Worker가 401 반환하고 이벤트 무시.

## 7. 보안

- Linear webhook 서명 시크릿: Worker 환경변수에 보관, 매 요청 HMAC 검증.
- 앱 ↔ relay 인증 토큰: Worker 환경변수 + 앱 로컬 설정에 보관.
- Linear API 토큰: **불필요**(webhook payload만으로 표시 가능, 클릭=닫기라 추가 조회 없음).

## 8. 테스트 전략

- `rule-engine`: 순수 함수이므로 다양한 이벤트/규칙 조합에 대한 단위 테스트.
- Worker 서명 검증: 유효/무효 서명에 대한 단위 테스트.
- WebSocket fan-out / 재연결 / 버퍼 replay: 통합 테스트 또는 수동 검증.

## 9. 범위에서 제외 (YAGNI)

- 멀티유저 / 멀티워크스페이스 (현재는 단일 사용자)
- 알림 클릭 시 Linear 열기 (클릭 = 닫기만)
- Linear API 토큰 기반 추가 조회
- 폴링 방식
- 규칙별 개별 소리/지속시간 설정 (전역 고정: 5초, 소리 ON)
